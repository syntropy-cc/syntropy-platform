# Prompt Index — Vision-to-System Framework

Reusable prompts for the complete software development lifecycle. Each prompt is designed to be copied into a new Cursor conversation. **No copy-paste of outputs between prompts is required** — the framework uses a persistent context file system to carry all information between sessions automatically.

For the **full framework guide** including philosophy, prerequisites, step-by-step walkthrough, and troubleshooting, see [`.cursor/FRAMEWORK.md`](../FRAMEWORK.md).

## Context Engineering System

The framework uses a **persistent context file system** to carry all information between prompt sessions. This eliminates all manual copy-paste of outputs.

Context files live in `docs/context/`. Every prompt reads what it needs from disk automatically and writes its outputs to disk before directing you to the next step.

### Context files

| File | Created by | Used by | What it contains |
|------|------------|---------|-----------------|
| `docs/context/EXECUTION-STATE.md` | 01-A (first write), all prompts (updates) | Every prompt | Current phase, last completed step, next step, registry of all other context files |
| `docs/context/architecture-brief.md` | 01-A | 01-B, 01-C | Confirmed Architecture Brief: domains, layer structure, documents to generate, ADR titles |
| `docs/context/architecture-file-list.md` | 01-B | 01-C | Complete list of all files generated by 01-B |
| `docs/context/generation-summary.md` | 01-C | 01-D, 01-E | Architecture generation summary: files created, domains, routing decision (01-D vs. 03) |
| `docs/context/ux-brief.md` | 01-D | 01-E | Confirmed UX Brief: interface types, DS-001 gate result, flows to design |
| `docs/context/ux-generation-summary.md` | 01-E | 01-F | UX generation summary: documents created, DS-001 gate result, design review result |

For full documentation of the context system, see `docs/context/README.md`.

## Plan-then-Execute Workflow

Every execute prompt (01-B, 01-C, 01-E, 01-F, 03, 04, 05) follows a two-step workflow:

1. **Plan mode** — the AI reads all context files and produces a full **Execution Plan** that includes:
   - **Context Loaded**: what was found in each file
   - **Scope**: files to create and estimated duration
   - **File Manifest**: every file with a content outline (not just a path — each row describes what goes in it)
   - **Key Decisions**: decisions that will be made during execution with preliminary conclusions
   - **Assumptions**: gaps in input context and what assumption covers them
   - **Execution Order**: files in dependency order with reasoning

2. **Confirm, then Agent mode** — you review the Execution Plan and confirm. Only after confirmation does the AI create any file. This gives you a meaningful review opportunity at every step.

**This workflow is enforced** — the AI will always wait for your confirmation before switching to execution. The plan itself is a deliverable, not a courtesy.

## How to Use

1. Run Prompt 01-A first — it assesses the Vision Document, produces the Architecture Brief, and saves it to `docs/context/architecture-brief.md`
2. For each subsequent prompt: open a **new** Cursor conversation, copy the prompt, and send
3. The AI automatically reads whatever context files it needs from `docs/context/` — no copy-paste required
4. Each prompt follows the **Plan-then-Execute** workflow: review the Execution Plan, confirm, then switch to Agent mode

**Analysis-only prompts** (01-A, 01-D): run in Cursor's Plan mode only. They produce no files other than context files.

**Execute prompts** (01-B, 01-C, 01-E, 01-F, 03, 04, 05): start in Plan mode to review the Execution Plan, then switch to Agent mode to execute.

## Lifecycle Order

```
Vision Document
    ↓
Prompt 01-A (Plan mode: assess Vision → produce Architecture Brief → save to docs/context/)
    ↓
Prompt 01-B (Plan → Agent: read architecture-brief.md → generate architecture docs → save file list)
    ↓
Prompt 01-C (Plan → Agent: read brief + file list → create ADRs + validation → save generation summary)
    ↓ (if user-facing interface exists — check generation-summary.md routing decision)
Prompt 01-D (Plan mode: read generation-summary.md → produce UX Brief → save to docs/context/)
    ↓
Prompt 01-E (Plan → Agent: read ux-brief.md + generation-summary.md → generate UX docs)
    ↓ (if web/dashboard interface — DS-001 gate was Required)
Prompt 01-F (Plan → Agent: read ux-generation-summary.md → generate Visual Direction + Image Prompts)
    ↓
Prompt 02 (repeat as needed for architecture iteration)
    ↓
Prompt 03 (Plan → Agent: read architecture → generate component records and backlog)
    ↓
Prompt 04 (Plan → Agent: read component records → generate Implementation Plan)
    ↓
Prompt 05 (Plan → Agent: read IMPLEMENTATION-PLAN.md → implement stage; repeat per session)
    ↓
Prompt 07 (generate user documentation)
    ↓
Deploy Docs Portal (manual step — see FRAMEWORK.md)
    ↓
Prompt 08 (evolve system — repeat as needed)
```

At any time, read `docs/context/EXECUTION-STATE.md` to see:
- Which phase the project is in
- Which prompt last ran
- What the next step is
- Whether any context files are missing or need attention

## Phase 8 Note

Phase 8 (Deploy Docs Portal) does not have an AI prompt. It is a manual step:

```bash
pip install mkdocs-material        # Install (one time)
mkdocs serve -f docs/mkdocs.yml    # Preview locally
mkdocs build -f docs/mkdocs.yml    # Build for deployment
```

See [FRAMEWORK.md](../FRAMEWORK.md) for deployment options (GitHub Pages, Netlify, Vercel, static hosting).
