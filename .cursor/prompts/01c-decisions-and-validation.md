# Architecture: Decisions and Validation (Prompt 01-C)

Use this prompt to record all significant architectural decisions as ADRs, initialize the architecture changelog, validate the generated architecture, and produce the final Generation Summary. This is **Step 3 of 3** in the architecture generation phase (Phase 2) of the Vision-to-System Framework.

**This prompt reads both required inputs from disk automatically.** The Architecture Brief is read from `docs/context/architecture-brief.md` and the file list from `docs/context/architecture-file-list.md`. No copy-paste from previous sessions is needed. If either file is missing, the prompt will halt.

After completing this prompt, the architecture phase is complete. If the system exposes any user-facing interface, proceed to **Prompt 01-D** (`.cursor/prompts/01d-ux-assess-and-brief.md`). Otherwise, proceed to **Prompt 03** (`.cursor/prompts/03-generate-implementation-docs.md`).

### What you must do (mandatory steps, in order)

#### Step 0 — Plan mode: read context and produce Execution Plan

Before creating any file, produce the full Execution Plan. Follow the standard format from `.cursor/templates/context/EXECUTION-PLAN-TEMPLATE.md`.

**Context Loaded** — summarize what you found:
- `docs/context/architecture-brief.md`: ADR working titles (list them), layer style chosen, domain count
- `docs/context/architecture-file-list.md`: total files created by 01-B, confirm key files are present (`ARCHITECTURE.md`, domain docs, diagram files)
- Architecture documents read: note any additional decisions observed in the generated documents that lack ADR coverage

**Scope** — number of ADRs to create, plus 1 changelog file, plus 1 Generation Summary context file.

**ADR Manifest** — the core deliverable of this Execution Plan. One row per ADR, including:

| # | ADR title | Decision subject | Key alternatives to consider |
|---|-----------|-----------------|------------------------------|
| 1 | ADR-001-{architecture-style} | Layer structure and style choice | [at least 2 alternatives from the Brief] |
| … | [from architecture-brief.md ADRs table] | [decision scope] | [alternatives to evaluate] |
| [any additional discovered] | [not in Brief but observed in generated docs] | [what decision it captures] | [alternatives] |

**Validation scope** — which architecture documents will be checked by the Architecture Validation Skill.

**Assumptions** — any gaps in context that require an assumption.

State: "Ready to create [N] ADRs. Confirm to proceed." Wait for confirmation (CTX-009).

#### Step 1 — Create ADR-001: Architecture Style

Create `docs/architecture/decisions/ADR-001-{architecture-style}.md` using the ADR template.

This ADR documents the layer structure and architectural style chosen in Prompt 01-A. It must include:
- **Context**: the forces that shaped the decision — team size and expertise from Vision Section 10, quality priorities from Section 9, delivery interfaces from Section 4, system complexity from Section 5
- **Decision**: the chosen architecture style, stated clearly ("We will use...")
- **Alternatives Considered**: at least two alternatives with pros, cons, and reason for rejection
- **Consequences**: positive (what is gained), negative (what trade-offs are accepted), neutral (side effects)
- **Derived Rules**: reference any rules in `.cursor/rules/architecture/architecture.mdc` that enforce this decision

#### Step 2 — Create additional ADRs

For each additional ADR in the Execution Plan, create a file in `docs/architecture/decisions/` using the ADR template.

Each ADR follows the same structure as ADR-001 (Context, Decision, Alternatives, Consequences, Derived Rules).

#### Step 3 — Initialize the Architecture Changelog

Create `docs/architecture/evolution/CHANGELOG.md` using the changelog template.

Add the initial entry documenting this generation session:
- **Type**: Added
- **Entry**: Record the full list of architecture documents and diagrams generated, grouped by category
- **Reference**: cite Prompt 01-A through 01-C as the generation method

#### Step 4 — Run Architecture Validation

Invoke the Architecture Validation Skill (`.cursor/skills/architecture-validation.md`) against the full `docs/architecture/` tree.

Execute every check the skill specifies. Do not simulate compliance — actually read the documents and verify each requirement. Record every finding, not just failures.

#### Step 5 — Write context outputs and present Generation Summary

1. Write `docs/context/generation-summary.md` using `.cursor/templates/context/GENERATION-SUMMARY-TEMPLATE.md`. Set `Status: Active`. Include:
   - Vision quality verdict and score (from architecture-brief.md)
   - Complete list of all documents created (across 01-B and 01-C)
   - Domains table with one-line responsibilities
   - Assumptions made
   - Architecture validation result
   - **Routing decision**: user-facing interfaces present (Yes/No), interface types, next step (01-D or 03)

2. Update `docs/context/EXECUTION-STATE.md` (CTX-002):
   - `Active phase`: Phase 2 — Architecture Generation (Complete)
   - `Last completed prompt`: 01-C — Decisions and Validation
   - `Next prompt`: 01-D — UX Assess and Brief [if user-facing interfaces] / 03 — Generate Implementation Docs [if no user-facing interfaces]
   - `Context File Registry`: set `architecture-brief.md` and `architecture-file-list.md` to `Delivered`; set `generation-summary.md` to `Active` (CTX-006)
   - `Completed Phases`: add entry for Phase 2 completion with ADR count, validation result, routing decision

#### Step 6 — Generate Initial LLM Artifact

Generate the initial `docs/llm/AGENTS.md` using the LLM documentation template and rules.

**Read before generating**:
- `.cursor/rules/documentation/llm-docs.mdc` — all rules (LLM-001 through LLM-017); follow every rule without exception
- `.cursor/templates/llm-docs/AGENTS-TEMPLATE.md` — fill this template top-to-bottom

**Sources to read** (already in context from Steps 1–5):
- `docs/architecture/ARCHITECTURE.md` → System Purpose → Section 1 (System Identity); technology stack, SDK info → Integration Prerequisites
- `docs/architecture/domains/*/ARCHITECTURE.md` → Data Architecture + ERD + API Design + Event Contracts + State diagrams → Sections 2, 3, 5
- `docs/architecture/platform/*/ARCHITECTURE.md` → environments, base URLs, CLI commands, API versioning → Integration Prerequisites and Section 3
- `docs/architecture/cross-cutting/*/ARCHITECTURE.md` → authentication flow (credential acquisition) → Integration Prerequisites and Section 4; all contracts → Section 4

**What to produce**:

Fill every section of the template. Set `completeness: architecture-only` in the header because code-phase details are not yet available.

**Critical**: the Integration Prerequisites subsection (at the top of Section 3) must NOT carry `TODO:` markers. Environment URLs, credential acquisition steps, and API versioning must come from the architecture. If the architecture does not specify these, write a note in the Generation Summary and ask the user to add them to the architecture before proceeding.

For other fields that require implementation details (specific error codes from code, concrete example values, implementation-observed failure modes), insert a marker:

```
TODO: populate at Prompt 07 — {specific detail needed}
```

Do not leave a section empty. If the architecture does not specify a contract (e.g., no rate limits are defined), state: `{Contract}: not applicable — {reason from architecture}`.

**Self-containment check before writing**: An agent building an integration must be able to make its first valid authenticated API call reading only `docs/llm/AGENTS.md`. No other directory (`docs/architecture/`, `docs/user/`, etc.) should be necessary. If any information required to make that first call is missing, add it inline or flag it explicitly in the Generation Summary as an architecture gap.

**Coverage check before writing**:
- Count every entity across all domain architecture docs → every one must appear in Section 2
- Count every API endpoint across all domain docs → every one must appear in Section 3
- Confirm cross-cutting docs cover authentication and error format → Section 4 must not be incomplete

**Token budget enforcement**:
After generating, estimate the token count. If it exceeds 6,000 tokens, move the lowest-priority domain sections to `docs/llm/AGENTS-EXTENDED.md` and add the overflow reference note at the bottom of `AGENTS.md` per the template.

**After writing `docs/llm/AGENTS.md`**, run the verification checklist from LLM-017.
