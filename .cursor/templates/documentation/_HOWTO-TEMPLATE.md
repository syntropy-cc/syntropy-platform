# How-to Guide Page Template

> How-to guides are **task-oriented**. They provide step-by-step directions for achieving a specific goal. Unlike tutorials, they assume the reader already has basic knowledge and wants to accomplish something specific.

## Template

```markdown
# How to {Accomplish Specific Task}

> **Goal**: {One sentence stating exactly what the reader will achieve}
>
> **Prerequisites**: {What the reader needs — keep minimal}

## Overview

{1-2 sentences. State the task and any important context. Do not teach — the reader knows the basics.}

## Steps

### 1. {Action Verb + What}

{Direct instruction. No lengthy explanation — just what to do.}

```{language}
{command or code}
```

### 2. {Action Verb + What}

{Continue with direct instructions.}

```{language}
{command or code}
```

### 3. {Action Verb + What}

{Continue...}

## Result

{Describe the expected outcome. Show output if applicable:}

```
{expected output}
```

## Variations

{Optional: show alternative approaches for common variations of this task.}

### {Variation Name}

{Brief description and adjusted steps or flags.}

```{language}
{alternative command or code}
```

## Troubleshooting

{Optional: list 1-3 common issues the reader might encounter.}

**{Problem description}**

{Solution.}

```{language}
{fix command if applicable}
```

## See Also

- [{Related Reference}](../reference/{page}.md)
- [{Related Tutorial}](../tutorials/{page}.md)
- [{Related How-to}](../how-to/{page}.md)
```

## Writing Guidelines

1. **Assume competence** — the reader knows the basics. Don't re-explain fundamentals.
2. **Be direct** — use imperative mood. "Run the command", not "You should run".
3. **Stay focused** — solve one task. If the guide covers multiple tasks, split it.
4. **Show variations** — real tasks have variations. Cover the common ones.
5. **Include troubleshooting** — anticipate where things go wrong.
6. **No teaching** — if the reader needs background, link to a tutorial or concept page.
